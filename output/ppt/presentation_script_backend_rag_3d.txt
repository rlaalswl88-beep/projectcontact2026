1. 발표 전체 메시지

이 프로젝트에서 제가 맡은 핵심은 설문 데이터를 단순히 저장하는 것에서 끝내지 않고, 사용자가 다시 확인할 수 있는 개인 결과, 전체 참여자의 통계, 그리고 사용자 메시지를 안전하게 노출하는 채팅 시스템까지 하나의 흐름으로 연결하는 것이었습니다.

구현 범위는 크게 네 가지입니다.

첫 번째는 MySQL 기반 데이터 연동입니다.
설문 참여자, 장면 메타데이터, 사용자 응답 데이터를 조인해서 개인 결과와 전체 통계 API를 만들었습니다.

두 번째는 쿠키 기반 개인화입니다.
로그인 없는 콘텐츠 특성을 고려해서 isolation_user_info 쿠키와 DB participant id를 연결하고, 사용자가 본인의 결과와 메시지를 다시 볼 수 있게 만들었습니다.

세 번째는 따뜻한 한마디 채팅의 RAG 검열입니다.
사용자 입력이 바로 화면에 노출되지 않도록 Trie 기반 금칙어 필터와 GPT-4o mini 문맥 검열을 결합했습니다.

네 번째는 Step2, Step3 인터랙션 시각화입니다.
Step2에서는 Three.js / React Three Fiber 기반 3D 심해 스크롤 경험을 만들었고, Step3에서는 참여자 수와 응답 비율을 모바일에서 읽기 쉬운 입자 애니메이션과 원그래프로 전환했습니다.


2. 슬라이드별 발표 문서

슬라이드 1. 프로젝트 개요

핵심 메시지:
이 프로젝트는 고립과 연결을 주제로 한 인터랙티브 콘텐츠이며, 제가 구현한 부분은 데이터가 실제 사용자 경험으로 이어지도록 만드는 풀스택 구조입니다.

발표 문장:
이 프로젝트는 사용자가 콘텐츠를 체험하고 설문에 응답한 뒤, 자신의 결과와 전체 통계를 확인하고, 마지막으로 따뜻한 메시지를 남길 수 있는 구조입니다.
제가 집중한 부분은 프론트 화면 자체보다 그 화면이 실제 DB 데이터와 연결되도록 만드는 백엔드, 통계 가공, RAG 검열, 그리고 3D 인터랙션입니다.

강조 포인트:
단순 화면 구현이 아니라 DB 저장, API 가공, AI 검열, 시각화까지 이어지는 흐름을 구현했습니다.


슬라이드 2. 문제 정의와 해결 방향

핵심 메시지:
문제는 데이터가 존재해도 사용자에게 다시 의미 있게 보여줄 구조가 없었다는 점입니다.

발표 문장:

DB 조인 기반 개인 결과 조회
연령대, 성별, 문항별 통계 집계

모바일 시연에 맞춘 통계 시각화

슬라이드 5. 전체 통계 API

핵심 메시지:
통계 API는 단순 count가 아니라, 연령대/성별/문항/답변 단위로 재가공한 데이터입니다.

발표 문장:

이후 user_responses와 scenes_metadata를 조인해서 어떤 장면의 어떤 질문에 어떤 응답이 얼마나 선택됐는지 계산했습니다.

기술적으로 어필할 점:
SQL에서 1차 그룹화를 하고, 서비스 계층에서 percentage와 프론트용 payload를 재구성했습니다.
프론트가 복잡한 DB 구조를 몰라도 graph, summary, scenes 구조만 받아 화면을 구성할 수 있게 만들었습니다.

슬라이드 7. RAG 하이브리드 검열 구조

핵심 메시지:
채팅 검열은 LLM 하나에만 의존하지 않고, 빠른 리스트 필터와 문맥 분석을 결합했습니다.

발표 문장:
따뜻한 한마디 채팅은 사용자가 직접 메시지를 남기는 기능이기 때문에 검열 로직이 필요했습니다.
먼저 fastscanner 기반 Trie 필터로 slang.csv와 리그오브레전드 필터링 리스트를 검사합니다.
이 1차 필터에서 닉네임이나 메시지가 걸리면 LLM을 호출하지 않고 즉시 FAIL 처리합니다.
1차 필터를 통과한 경우에만 messageRag.js의 시스템 프롬프트와 buildWarmFilterPrompt를 사용해 GPT-4o mini로 문맥 검열을 수행합니다.

기술적으로 어필할 점:
비용이 드는 LLM 호출 전에 로컬 Trie 필터를 먼저 적용해 성능과 비용을 줄였습니다.
단순 금칙어가 아니라 문맥상 공격성, 혐오, 부적절 표현까지 2차로 판단하도록 설계했습니다.

코드 캡처 연결:

슬라이드 8. 채팅 DB 상태 처리

핵심 메시지:
사용자 메시지는 바로 노출하지 않고 상태값으로 통제합니다.

발표 문장:
메시지가 등록되면 cheer_messages 테이블에 먼저 PENDING 상태로 저장합니다.
이후 비동기 moderateMessage 함수가 실행되고, 검열 결과에 따라 PASS 또는 FAIL로 업데이트합니다.
완전 통과하면 PASS, 닉네임만 위반이면 닉네임을 “따뜻한마음”으로 바꾸고 PASS, 메시지가 위반이면 FAIL로 처리해서 화면에서 숨깁니다.
LLM 호출 실패나 JSON 파싱 오류가 발생하면 기본값은 FAIL로 처리했습니다.

기술적으로 어필할 점:
사용자 응답 속도를 위해 검열 업데이트는 async/await 기반 비동기 처리로 분리했습니다.
외부 API 실패 시에도 부적절 메시지가 노출되지 않도록 fail-safe 정책을 적용했습니다.

코드 캡처 연결:
비동기 검열 파이프라인: output\ppt\img\08_warm_chat_async_moderation_pipeline.png

슬라이드 11. 모바일 3D 최적화

핵심 메시지:
데스크톱 연출은 유지하고 모바일에서만 렌더링 비용을 줄였습니다.

발표 문장:
QA 과정에서 Step2가 모바일에서 프레임드랍이 심하다는 문제가 있었습니다.
원인은 3D 버블, 고해상도 구체, Environment, CSS blur/filter, 자이로 이벤트가 모바일 GPU에 부담을 주기 때문이었습니다.
그래서 데스크톱 설정은 유지하고, 모바일일 때만 버블 수를 줄이고 sphereGeometry 해상도, dpr, antialias, Environment, CSS filter 비용을 낮췄습니다.
또 자이로 이벤트는 너무 자주 반응하지 않도록 약 20fps 수준으로 제한했습니다.

트러블슈팅 포인트:
문제: 모바일 Step2에서 프레임드랍 발생
원인: WebGL 렌더링 비용과 blur/filter, 자이로 이벤트가 동시에 발생
해결: 모바일 조건부 저비용 렌더링 경로 적용

4. 예상 질문과 답변

Q1. 왜 로그인 대신 쿠키를 사용했나요?
A. 이 콘텐츠는 공익적이고 가벼운 참여형 콘텐츠라 로그인이나 이메일/전화번호 수집이 사용자 이탈을 만들 수 있다고 판단했습니다. 그래서 isolation_user_info 쿠키와 participant id를 연결해 최소한의 개인화만 제공했습니다.

Q2. RAG 검열에서 왜 Trie 필터를 먼저 사용했나요?
A. 명확한 금칙어는 LLM까지 갈 필요가 없고, 로컬에서 빠르게 처리하는 것이 비용과 속도 면에서 유리합니다. LLM은 1차 필터를 통과한 메시지의 문맥 판단에만 사용했습니다.

Q3. LLM이 실패하면 어떻게 되나요?
A. 기본값은 FAIL입니다. 외부 API나 JSON 파싱이 실패해도 부적절한 메시지가 노출되지 않도록 fail-safe 정책으로 처리했습니다.

Q4. 3D 그래프를 왜 최종적으로 원그래프로 바꿨나요?
A. 3D 그래프는 시각적으로는 좋지만 모바일 세로 화면에서 정확한 비율을 읽기 어렵습니다. 그래서 참여자 수는 입자 애니메이션으로 직관적으로 보여주고, 문항별 비율은 원그래프로 명확하게 전달하도록 바꿨습니다.

Q5. 모바일 프레임드랍은 어떻게 해결했나요?
A. 데스크톱 연출은 유지하고 모바일일 때만 버블 수, 구체 해상도, dpr, antialias, Environment, CSS filter, 자이로 이벤트 빈도를 줄였습니다. 즉, 콘텐츠 방향은 유지하되 모바일 렌더링 비용만 낮췄습니다.


5. 발표 마무리 요약

제가 이 프로젝트에서 보여주고 싶은 역량은 세 가지입니다.

첫째, DB 데이터를 화면 요구사항에 맞게 재구성하는 백엔드 설계 역량입니다.
둘째, 사용자 입력을 안전하게 다루기 위한 RAG 기반 검열 흐름 설계 역량입니다.
셋째, 3D 인터랙션과 모바일 성능 사이에서 실제 사용자 경험을 기준으로 기술 선택을 조정한 점입니다.

이 프로젝트는 단순히 화면을 예쁘게 만든 작업이 아니라, 저장된 데이터와 AI 검열, 통계 시각화, 모바일 인터랙션을 하나의 서비스 흐름으로 연결한 풀스택 구현 경험입니다.
